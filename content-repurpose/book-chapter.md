# 4장. 콘텐츠 대량 생산 — "하나 만들면 열 개가 나오게"

> 유튜브 영상 1개 → 블로그 포스트 + SNS 게시물 + 뉴스레터 + 썸네일 가이드
> 소요 시간: 1.5시간 | 난이도: ★★☆
> 바로 쓸 수 있는 곳: 멀티 플랫폼 운영, 콘텐츠 리퍼포징, 마케팅 자동화

---

## 이 장에서 만드는 것

유튜브 대본 하나를 입력하면 12개 플랫폼에 최적화된 콘텐츠가 자동으로 생성되는 클로드 스킬을 만들어봅니다. 단순히 텍스트를 복사하는 게 아닙니다. 각 플랫폼의 글자 수 제한, 톤, 해시태그, CTA까지 모두 반영된 **즉시 게시 가능한 완성본**이 나옵니다.

이 장을 마치면 여러분은:
- 클로드 스킬의 구조(`SKILL.md`)를 이해하고 직접 만들 수 있게 됩니다
- 커스텀 슬래시 명령어(`.claude/commands/`)를 설계하는 방법을 배웁니다
- 만든 스킬을 zip으로 패키징해서 다른 프로젝트에서도 재사용할 수 있습니다

---

## 4.1 완성된 스킬의 전체 구조

먼저 우리가 만들 스킬의 파일 구조를 확인합니다.

```
content-repurpose/
├── SKILL.md                        ← 스킬 정의 파일 (핵심)
├── .claude/
│   └── commands/
│       └── repurpose.md            ← /repurpose 슬래시 명령어
├── examples/
│   └── example-usage.md            ← 사용 예시
└── README.md
```

각 파일의 역할을 정리하면 이렇습니다.

| 파일 | 역할 | 필수 여부 |
|------|------|----------|
| `SKILL.md` | 스킬의 두뇌. 클로드가 언제, 어떻게 동작할지 정의 | 필수 |
| `.claude/commands/repurpose.md` | `/repurpose` 명령어의 실행 로직 | 선택 |
| `examples/` | 사용 예시와 샘플 데이터 | 선택 |

**핵심 포인트:** 클로드 스킬에서 실제로 필요한 파일은 `SKILL.md` 단 하나입니다. 나머지는 확장을 위한 옵션입니다.

---

## 4.2 SKILL.md 작성하기 — 스킬의 두뇌

`SKILL.md`는 클로드 스킬의 핵심 파일입니다. 이 파일 하나에 "이 스킬이 무엇이고, 언제 작동하며, 어떻게 동작하는지"가 모두 담깁니다.

### YAML 프론트매터: 스킬의 신분증

파일 맨 위에 `---`로 감싼 YAML 블록이 프론트매터입니다. 클로드가 스킬을 인식하는 데 사용하는 메타데이터입니다.

```yaml
---
name: content-repurpose
description: 원본 콘텐츠(유튜브 대본, 강의, 회의록 등)를 여러 플랫폼에 최적화된 콘텐츠로 자동 변환합니다. 콘텐츠 리퍼포징, 감사, 갭 분석을 지원합니다.
---
```

여기서 주의할 점이 있습니다.

**`name` 필드 규칙:**
- 소문자, 숫자, 하이픈(-)만 사용 가능
- `Content Repurpose` → 에러 발생
- `content-repurpose` → 정상 작동
- 최대 64자

**`description` 필드 규칙:**
- 최대 200자
- 클로드가 "이 스킬을 언제 자동으로 호출할지" 판단하는 기준이 됩니다
- 구체적으로 적을수록 자동 트리거 정확도가 올라갑니다

> **실전 팁:** description은 "무엇을 하는가"뿐 아니라 "언제 사용하는가"를 함께 적으세요. "원본 콘텐츠를 변환합니다"보다 "유튜브 대본, 강의, 회의록 등을 여러 플랫폼에 최적화된 콘텐츠로 자동 변환합니다"가 더 정확하게 트리거됩니다.

### 본문 구조: 클로드에게 주는 상세 지시서

프론트매터 아래가 본문입니다. 마크다운으로 작성하며, 클로드가 실제로 따라야 할 행동 규칙을 정의합니다.

우리 스킬의 본문은 크게 4가지 블록으로 구성됩니다.

**블록 1 — 실행 규칙 (가장 먼저 읽히는 부분)**

```markdown
## 중요: 실행 규칙

- 이 스킬이 로드되면 외부 파일을 검색하거나 확인하지 말고
  바로 아래 프로세스를 실행하세요.
- "스킬 파일이 없다", "파일을 찾을 수 없다" 등
  내부 동작에 관한 말을 사용자에게 절대 하지 마세요.
- 이 SKILL.md 안의 지시만으로 충분합니다.
  추가 파일을 찾지 마세요.
```

이 블록이 왜 중요할까요? 클로드는 스킬을 실행할 때 습관적으로 관련 파일을 탐색하려 합니다. 파일이 없으면 "스킬 파일이 없네요"라고 사용자에게 말해버립니다. 이런 불필요한 메시지를 차단하려면 **본문 최상단에 명시적으로 금지 규칙을 적어야 합니다.**

> **실전 팁:** 클로드 스킬을 만들 때 가장 흔한 실수가 "클로드가 알아서 하겠지"라는 가정입니다. 원하지 않는 행동은 명시적으로 금지하세요. 특히 "~하지 마세요"가 "~하세요"보다 더 효과적입니다.

**블록 2 — 실행 모드 정의**

우리 스킬은 3가지 모드를 지원합니다.

| 모드 | 기능 | 사용 시점 |
|------|------|----------|
| 변환 (기본) | 원본 → 멀티 플랫폼 콘텐츠 | 가장 자주 사용 |
| 감사 (audit) | 기존 콘텐츠 라이브러리 분석 | 콘텐츠 전략 점검 시 |
| 갭 분석 (gap) | 경쟁 대비 빈 영역 파악 | 새 콘텐츠 기획 시 |

**블록 3 — 플랫폼별 변환 규칙**

각 플랫폼마다 글자 수, 톤, 필수 구조를 정의합니다. 이 부분이 스킬의 품질을 결정합니다.

예를 들어 X(Twitter) 스레드 규칙을 보면:

```markdown
#### X (Twitter) 스레드

- 1/N 형식의 스레드
- 각 트윗: 280자(한글 기준 140자) 이내
- 첫 트윗: 강력한 훅 (질문, 충격적 사실, 반전)
- 마지막 트윗: CTA + 리트윗 유도
- 총 5~10개 트윗
- 톤: 간결, 임팩트, 구어체
- 필수 구조: 훅 → 핵심 포인트 → 인사이트 → CTA
- 해시태그: 마지막 트윗에 2~3개
```

LinkedIn 게시물의 규칙은 완전히 다릅니다:

```markdown
#### LinkedIn 게시물

- 본문: 1,300자 이내 (약 200~250단어)
- 첫 3줄: 강력한 훅 (접히기 전에 보이는 영역)
- 단락: 1~2문장씩 짧게, 줄바꿈 적극 활용
- 톤: 전문가적, 인사이트 중심, 경험 공유형
- 필수 구조: 훅 → 경험/스토리 → 인사이트 → 질문형 CTA
- 해시태그: 본문 하단 3~5개
```

같은 원본이라도 X에서는 140자짜리 임팩트 있는 문장이, LinkedIn에서는 전문가 톤의 긴 글이 나옵니다. 이 차이를 만드는 게 바로 플랫폼별 규칙입니다.

우리 스킬이 지원하는 전체 12개 플랫폼은 다음과 같습니다.

| 플랫폼 | 글자 수 | 톤 | 핵심 구조 |
|--------|---------|-----|----------|
| SEO 블로그 | 1,500~2,500자 | 정보 전달형 | H2/H3 + 키워드 + CTA |
| X 스레드 | 트윗당 140자(한글) | 간결, 임팩트 | 훅 → 포인트 → CTA |
| LinkedIn | 1,300자 이내 | 전문가적 | 훅(3줄) → 인사이트 → 질문 |
| Instagram | 2,200자 이내 | 친근, 감성 | 훅 → 본문 → CTA → 해시태그 |
| 뉴스레터 | 500~800단어 | 1:1 대화형 | 인사 → 핵심 → CTA |
| 썸네일 가이드 | - | 시각적 | 텍스트 + 색상 + 레이아웃 |
| YouTube Shorts | 150~300자 | 빠르고 에너지 | 3초 훅 → 본론 → CTA |
| TikTok | 150~300자 | 빠르고 에너지 | 3초 훅 → 본론 → CTA |
| Facebook | 500자 이내 | 커뮤니티형 | 훅 → 본문 → 의견 요청 |
| Pinterest | 500자 이내 | 실용적 | 키워드 설명 + 행동 유도 |
| Threads | 500자 이내 | 캐주얼, 트렌디 | 훅 → 짧은 본문 → 반응 유도 |
| 브런치 | 2,000~4,000자 | 에세이형 | 감성 도입 → 본론 → 여운 |

**블록 4 — 출력 규칙 (결과물의 품질을 결정하는 핵심)**

스킬을 만들 때 가장 신경 써야 하는 부분입니다. 아무리 좋은 콘텐츠를 생성해도, 출력 형식이 복사-붙여넣기에 적합하지 않으면 쓸모가 없습니다.

```markdown
### 4단계: 출력

**핵심 원칙: 각 플랫폼 결과물은 사용자가 그대로 복사하여
해당 플랫폼에 붙여넣기만 하면 바로 게시할 수 있는 완성본이어야 합니다.**

아래 출력 규칙을 반드시 따르세요:

1. 플랫폼별로 구분: ## [플랫폼명] 헤더 바로 아래에 완성된 콘텐츠를 출력합니다.
2. 복사-붙여넣기 최적화:
   - "[여기에 입력]", "[링크]", "[이미지]" 같은 플레이스홀더를
     절대 사용하지 마세요. 실제 내용으로 채우세요.
   - "톤:", "형식:", "필수 요소:" 같은 메타 레이블을
     결과물에 넣지 마세요.
   - 코드 블록으로 콘텐츠를 감싸지 마세요.
   - "아래는 ~입니다" 같은 설명 문구를
     콘텐츠 앞뒤에 붙이지 마세요.
```

이 규칙을 넣지 않으면 클로드는 높은 확률로 이런 결과를 냅니다:

```
[톤: 전문가적]
[형식: LinkedIn 게시물]

아래는 LinkedIn 게시물로 변환한 결과입니다:

"""
(콘텐츠)
"""
```

사용자는 이걸 매번 수동으로 다듬어야 합니다. 출력 규칙을 명확히 적으면 바로 쓸 수 있는 깔끔한 결과물이 나옵니다.

> **실전 팁:** "~하지 마세요" 규칙은 구체적인 **나쁜 예시**를 함께 적으면 효과가 훨씬 좋습니다. 클로드가 정확히 어떤 패턴을 피해야 하는지 이해합니다.

## 4.3 슬래시 명령어 만들기 — /repurpose

스킬(`SKILL.md`)은 클로드가 자동으로 판단해서 호출합니다. 하지만 명시적으로 "지금 이 기능을 실행해"라고 말하고 싶을 때가 있습니다. 그때 쓰는 것이 **슬래시 명령어**입니다.

### 슬래시 명령어의 위치

```
.claude/
└── commands/
    └── repurpose.md    ← 이 파일이 /repurpose 명령어가 됩니다
```

파일 이름이 곧 명령어 이름입니다. `repurpose.md` → `/repurpose`

### 스킬 vs 슬래시 명령어: 차이점

| 구분 | 스킬 (SKILL.md) | 슬래시 명령어 (.claude/commands/) |
|------|-----------------|-------------------------------|
| 호출 방식 | 클로드가 자동 판단 | 사용자가 `/명령어`로 직접 호출 |
| 위치 | 프로젝트 루트 또는 스킬 업로드 | `.claude/commands/` 디렉토리 |
| 인자 전달 | 자연어로 처리 | `$ARGUMENTS` 변수로 전달 |
| 활용 | 폭넓은 자동 트리거 | 특정 워크플로우의 명시적 실행 |

### /repurpose 명령어의 구조

```markdown
# /repurpose - 원소스 멀티콘텐츠 자동 생성

원본 콘텐츠 하나를 여러 플랫폼에 최적화된 콘텐츠로
자동 변환하는 통합 명령어입니다.

## 사용법

/repurpose                    → 변환 모드
/repurpose [원본 텍스트]       → 변환 모드 (텍스트를 원본으로 사용)
/repurpose audit              → 콘텐츠 감사 모드
/repurpose gap                → 콘텐츠 갭 분석 모드
```

핵심은 `$ARGUMENTS` 변수입니다. 사용자가 `/repurpose` 뒤에 입력한 모든 텍스트가 이 변수에 들어갑니다.

```markdown
### 모드 판별

$ARGUMENTS 값을 확인합니다:
- `audit` 또는 `감사`로 시작하면 → 감사 모드 실행
- `gap` 또는 `갭`으로 시작하면 → 갭 분석 모드 실행
- 그 외 텍스트가 있으면 → 변환 모드 (해당 텍스트를 원본으로 사용)
- 비어 있으면 → 변환 모드 (원본 입력을 요청)
```

이렇게 하나의 명령어 안에서 인자에 따라 모드를 분기하면, `/repurpose`, `/content-audit`, `/content-gap` 세 개의 명령어를 따로 만들 필요가 없습니다. 사용자 입장에서도 하나만 기억하면 됩니다.

### 대화형 플로우: 부족한 정보는 질문으로 채운다

슬래시 명령어 파일에는 **대화형 플로우**도 정의할 수 있습니다. 클로드가 `AskUserQuestion` 도구를 사용해서 사용자에게 선택지를 제시하는 방식입니다.

```markdown
**3단계: 플랫폼 선택**

AskUserQuestion 도구를 사용하여 multiSelect: true로
플랫폼을 선택받습니다:

질문: "어떤 플랫폼용 콘텐츠를 생성할까요? (복수 선택 가능)"
옵션:
- SEO 블로그 포스트
- X (Twitter) 스레드
- LinkedIn 게시물
- Instagram 캡션
```

실행하면 클로드가 깔끔한 선택 UI를 보여줍니다. 사용자는 체크박스를 클릭하듯 원하는 플랫폼을 고르면 됩니다. 타이핑할 필요 없이 클릭만으로 진행됩니다.

## 4.4 실전 사용: 유튜브 대본 → 3개 플랫폼 변환

실제 사용 과정을 처음부터 끝까지 살펴봅시다.

### Step 1: 명령어 실행

클로드 코드에서 다음과 같이 입력합니다.

```
/repurpose
```

### Step 2: 원본 콘텐츠 입력

클로드가 원본을 요청합니다. 유튜브 대본을 붙여넣습니다.

```
안녕하세요, 오늘은 1인 개발자가 실제로 수익을 만드는
현실적인 방법 3가지를 알려드리겠습니다...
```

### Step 3: 분야와 톤 선택

클로드가 질문합니다:

> 이 콘텐츠의 분야는 무엇인가요?
> → **테크/개발** 선택

> 원하는 톤/말투가 있나요?
> → **캐주얼** 선택

### Step 4: 플랫폼 선택

> 어떤 플랫폼용 콘텐츠를 생성할까요? (복수 선택 가능)
> → **SEO 블로그 포스트**, **X 스레드**, **Instagram 캡션** 선택

### Step 5: 결과 확인

## 4.5 감사 모드와 갭 분석 모드

콘텐츠를 변환하는 것 외에도 전략적 분석 기능이 있습니다.

### 콘텐츠 감사: /repurpose audit

기존에 만들어둔 콘텐츠 목록을 분석합니다. 어떤 주제가 부족한지, 어떤 콘텐츠를 업데이트해야 하는지, 중복되는 건 없는지 한눈에 파악할 수 있습니다.

```
/repurpose audit
```

클로드가 콘텐츠 목록을 요청하면 제목, 주제, 날짜가 포함된 리스트를 붙여넣습니다. 그러면 다음과 같은 분석 결과가 나옵니다:

- **주제별 분포 표**: 어떤 주제에 콘텐츠가 몰려 있는지
- **부족한 주제 영역**: 아직 다루지 않은 중요 주제
- **업데이트 필요 콘텐츠**: 오래되어 정보가 낡은 글
- **중복/통합 가능 콘텐츠**: 비슷한 주제의 글을 하나로 합칠 수 있는 경우
- **우선순위별 권장 액션**: 지금 당장 해야 할 것부터 순서대로

### 갭 분석: /repurpose gap

경쟁 채널과 비교해서 우리에게 부족한 영역을 찾습니다.

```
/repurpose gap
```

클로드가 세 가지를 물어봅니다:
1. 현재 보유한 콘텐츠 주제/목록
2. 비교할 경쟁 채널이나 사이트
3. 분석하고 싶은 범위

분석 결과로 경쟁 대비 미보유 주제, 형식 갭(경쟁사는 인포그래픽을 활용하지만 우리는 안 하고 있다 등), 플랫폼 갭, 그리고 콘텐츠 생성 로드맵까지 제시합니다.

## 4.6 스킬 패키징과 배포

### zip 파일 규칙

완성된 스킬을 다른 프로젝트에서 사용하거나 공유하려면 zip으로 패키징합니다. 이때 반드시 지켜야 할 규칙이 있습니다.

**올바른 구조:**
```
content-repurpose.zip
└── content-repurpose/          ← 최상위 폴더 필수
    ├── SKILL.md                ← 필수
    └── (기타 파일들)
```

**잘못된 구조:**
```
content-repurpose.zip
├── SKILL.md                    ← 최상위 폴더 없이 바로 파일 (에러)
└── (기타 파일들)
```

흔히 하는 실수 세 가지를 정리합니다.

| 실수 | 에러 메시지 | 해결 |
|------|-----------|------|
| 최상위 폴더 없음 | "All files must be inside the top-level folder" | zip 안에 폴더를 만들고 그 안에 파일 배치 |
| SKILL.md 없음 | "Zip must contain a SKILL.md file" | 반드시 SKILL.md 파일 포함 |
| name에 대문자 사용 | "Skill name can only contain lowercase letters, numbers, and hyphens" | 소문자와 하이픈만 사용 |

### zip 만드는 명령어

```bash
cd 프로젝트_상위_디렉토리
zip -r content-repurpose.zip content-repurpose/ -x "*.DS_Store"
```

### 업로드

클로드 코드 설정 → 스킬(Skills) → 스킬 업로드에서 zip 파일을 드래그 앤 드롭합니다.

## 4.7 나만의 스킬로 커스터마이징하기

이 스킬은 시작점입니다. SKILL.md를 수정해서 자신의 업무에 맞게 바꿀 수 있습니다.

### 커스터마이징 예시

**교육 유튜버라면:**
- 플랫폼 목록에 "퀴즈", "학습 가이드 PDF", "요약 노트" 추가
- SEO 블로그 규칙에 "학습 목표를 도입부에 명시" 추가
- CTA를 "다음 강의 수강하기"로 변경

**뷰티 유튜버라면:**
- 플랫폼 목록에 "제품 리스트", "스텝별 가이드", "쇼핑 링크 모음" 추가
- Instagram 규칙에 "사용 제품을 태그 형식으로 나열" 추가
- 썸네일 가이드에 "비포/애프터 레이아웃 기본" 추가

**비즈니스 회의라면:**
- 입력 형태를 "회의록"으로 변경
- 플랫폼 목록을 "액션 아이템", "이메일 요약", "Notion 페이지"로 교체
- 톤을 "비즈니스 공식" 고정

SKILL.md의 플랫폼별 규칙 섹션만 수정하면 됩니다. 전체 구조를 바꿀 필요는 없습니다.
